# 旅行プランナーシステム 仕様書

## プロジェクト概要

旅行先と日程を入力すると、その旅行に必要な**すべての情報**を1つのWebサイトにまとめるシステム。
このサイトだけ持っていけば旅行中に他に何も要らない、という完成度を目指す。

初回のケーススタディとして**インドネシア（ジャカルタ → バリ島フェリー乗り場）10日間**の旅行プランを作成する。
このケーススタディを通じて、再利用可能なスキルとエージェントを構築し、将来の旅行（フィリピン、タイ等）にも同じシステムで対応できるようにする。

---

## 第1回旅行プラン：インドネシア

### ルート概要

東京（成田/羽田）→ ジャカルタ着 → 陸路で東へ移動 → イジェン火山（硫黄・ブルーファイア）→ バリ島行きフェリー乗り場（ケタパン/ギリマヌク港）着で終了。

### 日程
- 全10日間
- イジェン火山は**必須**（硫黄の青い炎を見たい）

### 旅行者のプロフィール
- 東京在住
- インドネシアは初めて（想定）
- 都会的な場所は求めていない

### 求める体験
- **最優先**: 南国インドネシアらしい雰囲気。ビルや都市ではなく、自然・海・ビーチ・人が少ない場所
- **必須**: イジェン火山（Kawah Ijen）の訪問
- ローカルな食事体験
- 観光客だらけでない本物のインドネシア
- ジャカルタは経由地として最小限にとどめ、なるべく早く自然豊かなエリアへ移動

### ルート設計の方針
- ジャカルタから東ジャワ方面へ移動する途中で、面白い立ち寄りスポットを含める
- バンドン、ジョグジャカルタ（ボロブドゥール）等の文化的スポットも候補
- 最終目的地はバリ島へのフェリー乗り場（ケタパン港またはギリマヌク港）
- イジェン火山はフェリー乗り場の近くなので、旅の終盤に配置

## サイトに含める情報（各スポット・各日程に対して）

### 必須情報

1. **アクセス方法**
   - 前のスポットからの移動手段（バス、電車、タクシー、配車アプリ、バイクレンタル等）
   - 所要時間
   - 費用（ルピア + 日本円換算の目安）
   - 具体的な手順（例：「Grabアプリで〇〇駅から△△まで」）

2. **そのエリアの街の情報**
   - 歩いて回れるか、移動手段が必要か
   - ATM・両替所の場所
   - WiFi/SIM事情
   - 治安（特に夜間）
   - コンビニ・スーパーの有無

3. **観光スポット**
   - 各スポットの詳細説明
   - 入場料・営業時間
   - ベストな訪問時間帯
   - 滞在目安時間
   - 注意事項・Tips（「靴が滑りやすい」「日焼け対策必須」等）
   - 行き方（そのエリアの中心部からの距離・手段）

4. **食事**
   - ローカルフード紹介（その地域の名物）
   - おすすめレストラン/ワルン/屋台（予算帯別：安い/中/高い）
   - 各店の人気メニュー、予算感
   - 「これを食べろ」という一品推薦
   - 飲み物のおすすめ（ローカルコーヒー、ジュース等）
   - 食の安全性に関する注意

5. **買い物・お土産**
   - 何が有名か
   - 相場感
   - おすすめの店・マーケット

6. **宿泊情報**
   - エリアごとのおすすめ宿（予算帯別）
   - 予約方法のTips

7. **実用情報（旅行全体で1ページ）**
   - ビザ・入国手続き
   - 通貨（ルピア）と両替の注意点
   - SIMカードの買い方とおすすめキャリア
   - 電圧・プラグ形状
   - 緊急連絡先（大使館、警察、病院）
   - 持ち物チェックリスト
   - インドネシア語の基本フレーズ

## UI/UXの要件

### デザイン原則
- **AIらしさを徹底排除** — 生成AI感のある表現、AIスタンプ、「AIが作りました」的な雰囲気は一切出さない
- 人間が丁寧に作った旅行ガイドブックのようなクオリティ
- スマートフォンファーストのレスポンシブデザイン
- 旅行中に片手で操作できるUI

### ビジュアル
- **画像を大量に使用** — 各スポットの実際の写真（ブログ記事、旅行サイトからの参照画像URLを使用）
- ヒーロー画像（各エリアのメインビジュアル）
- 食事の写真
- 地図・ルート図

### 画面構成

#### トップページ
- 旅行の概要（ルートマップ、日程サマリー）
- 日別のカード一覧（Day 1, Day 2, ... Day 10）
- 実用情報へのリンク

#### 日別ページ（各Day）
- その日のルート概要
- タイムライン形式（朝→昼→夜）
- 各スポットの詳細カード（タップで展開）
- その日の食事おすすめ
- 移動情報

#### 詳細カード（各スポット）
- 写真
- 説明文
- アクセス方法
- 料金・時間
- Tips
- Google Maps リンク

#### 実用情報ページ
- ビザ、通貨、SIM、緊急連絡先等

### 技術仕様
- React（.jsx）で構築
- GitHub Pages でホスティング
- オフライン対応を検討（Service Worker / PWA）
- データは JSON ファイルで管理（data/indonesia-jakarta-to-bali.json 等）

## 品質管理プロセス（最重要）

### 反復改善ループ

このプロジェクトの核心は**作って終わりではなく、何度もレビュー→改善を繰り返す**こと。

#### 第1イテレーション：初版作成
- ルート設計、各日程の骨格を作成
- 基本的な観光スポット・食事情報を埋める
- UIの初版を生成

#### 第2イテレーション：ユーザー視点レビュー
レビュアーは**以下の立場で**初版を見る：
- 「初めてインドネシアに行く日本人旅行者」として
- 「このサイトだけを頼りに実際に旅行する人」として

チェック項目：
- この情報だけで実際に移動できるか？曖昧な表現はないか？
- 「〇〇がおすすめ」とだけ書いて、行き方や料金が書いてないスポットはないか？
- 画像がなくてイメージが湧かない場所はないか？
- 食事情報が薄いエリアはないか？
- 予算感がわからない項目はないか？
- 「で、結局どうすればいいの？」と思う箇所はないか？

#### 第3イテレーション：情報補完
- レビューで指摘された不足情報をWeb検索で追加調査
- 画像の追加
- 具体的な金額・時間の補記
- 実体験に基づくブログ記事等からのTips追加

#### 第4イテレーション以降：繰り返し
- 必要に応じてさらにレビュー→補完を繰り返す
- 最低3回のイテレーションを行う

## 開発アプローチ（Option C）

### Phase 1：ウブド相当の1エリアを手動で作る
- ジャカルタ到着初日、もしくはルート上の最初の自然エリア1つだけを作る
- この過程で「何を調べたか」「どういう構造が必要だったか」「何が大変だったか」を記録
- UIの基本デザインもこの段階で確定

### Phase 2：Phase 1の経験からスキルを抽出
- travel-research スキル：調査の手順書
- travel-review スキル：レビューの基準書
- travel-ui スキル：UI生成のルール

### Phase 3：スキルを使って次のエリアを生成→検証
- 作ったスキルで別のエリアを自動生成
- 結果を確認してスキルを改善

### Phase 4：必要に応じてエージェント化
- コンテキストが足りなくなったら分離
- 食事専門、交通専門など

### Phase 5：全エリア展開（Agent Teams）
- 改善されたスキル・エージェントで残り全エリアを生成
- 反復レビューループを実行

### Phase 6：デプロイ
- GitHub Pages に公開
- スマホでの表示確認
- オフライン対応

## 非機能要件

- **パフォーマンス**: スマホの3G回線でも3秒以内に表示
- **オフライン**: 可能な限りオフラインでも閲覧できる設計
- **多言語**: 初版は日本語のみ。将来的に英語対応も視野に
- **更新性**: JSONデータを差し替えるだけで内容更新が可能な設計
- **再利用性**: 国名・エリア名を変えるだけで別の旅行にも使えるフレームワーク
